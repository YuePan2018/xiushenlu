# AGENTS

本文件给后续参与本项目的 agent 或开发者使用。目标是先理解项目方向、边界和协作习惯，再动代码。

## 先读

- `README.md`：项目介绍、demo 流程和 milestone。
- `TECHNICAL.md`：运行命令、控制台说明、配置、模块职责、数据规则和验证建议。
- `docs/规划/2026-05-08_修身炉知识成长助手规划.md`：最新知识成长助手规划。

如果这三份文档口径冲突，以最新规划和本文件的安全边界为准。

## 项目定位

修身炉是一个本地优先的知识成长助手。

核心目标不是自动替用户做所有事，而是和用户一起消化知识、积累资料、运用知识，并把过程变成可追踪、可复用、可继续升级的个人知识系统。

```text
资料进入 -> 消化整理 -> 知识沉淀 -> 运用反馈 -> 再次升级
```

所有 pipeline、post、工具、通知、自动化和审批能力，都应服务于这条主线。工具不是目的，自动化也不是目的；它们的价值在于帮助积累资料、加工资料、调用知识和验证知识。

当前仍不做完整自主 agent。第一原则保持：

```text
代码控制流程，LLM 负责总结、整理、建议和结构化草稿。
```

近期不要急着扩大自主性。先稳定三件事：

```text
接住资料、消化资料、用回知识。
```

## 工作原则

- 先资料、再知识、再自动化。
- 先结构清楚，再追求检索智能。
- 先人工可读，再机器可用。
- 先草稿、再审批、再写入权威记忆。
- 今日待办保持统一 `【分组】` 加编号列表；优先级只出现在计划里的“任务管理”表，不写回 today_tasks。
- daily review 滚动待办时，只能来自计划内未完成任务和显式明日计划；过程记录只作为证据和复盘分析，不自动派生第二天任务。
- 不让 LLM 输出直接变成 shell 命令。
- 不让 LLM 直接对外发布内容。
- 不轻易引入自主 agent loop；只有工具注册、审批、预算和暂停机制稳定后，再扩大自主性。
- 项目内 Codex skill 放在 `.codex/skills/` 并由 Git 追踪；技术储备 skill 以项目内 `.codex/skills/xiushenlu-tech-reserve/` 为准。
- 更新功能时，优先同步 `README.md`、`TECHNICAL.md` 和相关规划或执行文档，但不要把技术手册内容重新塞回 `AGENTS.md`。
- `README.md` 保持项目定位、能力概览、面向用户的主流程和关键入口；不要放配置项、命令细节、第三方仓库布局、参数说明、实现机制或排障步骤。
- README 末尾已有统一的 `TECHNICAL.md` 技术细节入口；新增功能段落不要反复写“详见 TECHNICAL”这类重复跳转。
- 技术细节统一放进 `TECHNICAL.md`，包括运行命令、配置路径、模块职责、第三方工具/MCP 适配、API/CLI 参数、验证命令和故障处理。

## 安全与数据边界

- 文件读写应优先使用 `app.safety.safe_read_text`、`safe_write_text`、`safe_append_text` 或已封装模块。
- 不要绕过 `app/safety.py` 直接写 `data/` 下的运行文件。
- 受保护路径：`docs/`、`data/memory/`。创建、修改、移动、删除这些路径下的文件前，必须先说明拟改文件、修改原因和预计改法，并获得用户明确同意；未获同意时只能读取和分析。
- 例外：`docs/吸纳/技术储备/` 是技术储备自由读写目录，可由技术储备 skill 创建、修改和维护索引，不需要按 `docs/` 受保护路径额外审批。
- 不自动读取浏览器、聊天软件、密钥、cookie 或项目外敏感目录。
- 不自动删除文件；需要清理时优先移动到 `data/quarantine/`。
- `data/` 下真实个人数据和运行数据默认不提交到 Git。
- `data/memory/goals.md` 是用户维护的长期目标权威来源，普通 pipeline 只能读取，不能自动改写。
- 知识条目要保留来源；不要把未核实资料写成确定结论。
- 私密 daily 原文、日志路径、密钥、聊天内容或敏感上下文不得直接用于公开输出。

## 开发与验证

- 运行命令、配置、模块职责、数据规则和验证建议见 `TECHNICAL.md`。
- 需要检索文件名、目录或内容时，默认使用 PowerShell 原生命令，不使用 `rg` / ripgrep。
- 读取中文文本文件时，显式使用 `-Encoding utf8`，例如 `Get-Content -Raw -Encoding utf8`、`Select-String -Encoding utf8`。
- 运行可能输出中文的 Python / conda 命令时，先设置 `$env:PYTHONUTF8 = "1"` 和 `$env:PYTHONIOENCODING = "utf-8"`；使用 `conda run` 时优先加 `--no-capture-output`。
- 后续 agent 或自动化运行 Python/验证命令时，默认使用 `conda run -n xiushenlu python ...`，不要直接使用裸 `python ...`。
- 不要假设 `conda activate xiushenlu` 会跨工具调用或跨 shell 生效；如果使用 `conda activate`，必须和实际 Python 命令放在同一条 PowerShell 调用中。
- 文档改动通常不需要跑 Python 测试；代码或 pipeline 改动按 `TECHNICAL.md` 的验证建议执行。

## Git子仓库
- `data/`、`config/`、`docs/`都是独立子仓库，同时也包含受保护路径。git操作必须分清在子仓库还是根仓库。
